1. 为什么我们需要量化

大语言模型的推理成本问题是所有落地场景绕不开的坎。一个70B参数的模型用FP16加载需要140GB显存——这还不包括推理时生成的KV Cache。在长上下文场景下，KV Cache本身就能再吃掉几十GB显存。

量化技术的核心目标非常直接：用更少的比特表示每个数，在精度损失可接受的前提下，降低内存占用、提升推理速度。它不是什么"黑魔法"，本质上是在做信息压缩——但和通用压缩不同，量化后的模型需要能直接进行计算，不需要先解压回高精度。

💡 关键认知

量化不是单纯"把数改小"。LLM的权重和激活分布有极强的结构性特征，大多数权重值集中在很窄的范围内，离群值（outlier）占比极低但极其重要。好的量化方案本质上是在识别并保护这些重要信息，同时对平凡信息做更激进的压缩。

2. 量化的数学基础

最基础的线性量化可以用一个简单公式描述：将浮点值 x 映射为整数 x_q：

x_q = round( x / s ) + z

反量化时：

x ≈ s · (x_q - z)

其中两个核心参数：

  • s (scale)：缩放因子，决定量化的步长。s = (x_max - x_min) / (2^b - 1)，b为比特数
  • z (zero-point)：零点偏移，处理非对称分布。对称量化时z=0，可以省掉一次减法计算

这就是为什么对称量化在硬件上更友好——反量化只需要一次乘法。代价是如果分布不对称，会浪费一部分动态范围。

量化误差来自哪里？

  1. 舍入误差：round操作带来的±0.5步长误差，这是不可避免的
  2. 截断误差：超出[x_min, x_max]范围的值被clamp，这是误差的主要来源，尤其是离群值存在时
  3. 误差累积：神经网络是多层叠加的，单层误差看似不大，深层累积后可能造成输出分布偏移

所有先进的量化算法，本质上都是在用不同的方式处理离群值和误差分配这两个问题。

3. 量化技术分类体系

当前工业界和学术界的量化方案可以清晰地分为几个方向，各自解决不同的问题：

类别 量化对象 精度 速度提升 精度损失 成熟度
W8A8 权重+激活 INT8 ~2x 极小（可做到几乎无损） ✅ 工业级成熟
W4A16 仅权重 INT4 ~1.5-2x (内存受限场景) 小（好方案几乎感知不到） ✅ 工业级成熟
KV Cache量化 KV Cache INT4/INT8 2-4x (长上下文) 中等（需仔细调优） 🔶 快速成熟中
Sub-4bit 权重 2-3bit ~3-4x 较大（需算法补偿） 🔬 研究阶段
QAT 权重+激活 任意 同PTQ 最小 🔶 部分场景落地

3.1 W8A8：权重+激活都量化

W8A8是理论上最理想的量化方案——因为矩阵乘法的两个输入都是INT8，可以直接调用Tensor Core的INT8算力，真正实现计算加速，而不仅仅是"省内存"。

但问题出在激活上。LLM的激活分布高度非均匀，存在大量跨通道离群值——某些通道的激活值比其他通道大100倍以上。如果直接做per-tensor量化，这些离群值会迫使scale设得很大，导致其他通道的量化精度完全崩溃。

SmoothQuant ICLR 2023

核心洞察：难量化的是激活，容易量化的是权重。离群值只出现在少数固定通道上，这一属性在模型各层是一致的。

核心技巧：引入一个逐通道缩放因子 s，在量化前做等价变换：Y = (X diag(s)) · (diag(s)^{-1} W)。通过选择合适的s，把激活的离群值"迁移"一部分到权重上——让激活的分布变平滑，权重本来就好量化，迁移一点不影响。

效果：7-70B模型上实现W8A8几乎无损，不需要微调，不需要重训，是目前工业界部署INT8推理的事实标准方案之一。

LLM.int8() NeurIPS 2022

核心思路：混合精度分解——把激活中超过阈值的离群特征单独拎出来用FP16计算，其余99%以上的特征用INT8计算。

观察：离群值只存在于大约30%的序列token、不到30%的隐藏维度中，单独处理这部分的计算开销很小，但能保住精度。

缺点：需要动态判断哪些是离群值，kernel实现复杂，相比纯INT8速度优势不明显。

3.2 W4A16：仅权重量化——当前端侧部署主流

W4A16是目前端侧和开源社区最火的方案：只把权重量化到INT4，激活还是用FP16。因为LLM推理是典型的内存带宽受限场景（Memory-Bound），权重从16bit降到4bit直接把内存带宽需求砍到1/4，推理速度自然就上来了——不需要INT8矩阵乘法加速，光"读权重更快"这一点就能带来1.5-2倍的速度提升。

这也是为什么llama.cpp、GPTQ、AWQ、MLX全部都首先支持W4A16——它的工程收益最高，实现难度相对可控，精度损失可以做得非常小。

GPTQ ICLR 2023

思想来源：把经典的OBS（Optimal Brain Surgeon）二阶剪枝方法改造成量化算法。

核心算法：按列顺序量化权重，每量化一列权重，就更新所有未量化权重来补偿这一步带来的误差。具体来说，利用Hessian逆矩阵计算最优的量化值和更新量：

for each column i in W:
    quantize column i to w_q
    compute quantization error δ = w_i - w_q
    update remaining columns: W[:, i+1:] -= δ * (H^{-1}[i+1:, i] / H^{-1}[i, i])

关键优化：原始OBS是O(n^3)，GPTQ通过Cholesky分解和批量更新把复杂度降到O(n^2)，可以在几小时内量化完70B模型。

特点：基于校准数据做全局误差最小化，精度非常好。缺点是量化速度相对慢，需要校准数据集。

AWQ (Activation-aware Weight Quantization) MLSys 2024

核心洞察：权重的重要性不是均等的——和大激活相乘的权重更重要。与其保护"权重大的"，不如保护"乘以大激活的"权重，因为它们对输出的贡献更大。

核心观察：只需保护大约1%的" salient weight"（重要权重），其余99%量化到4bit几乎不损失性能。

实现方式：不做复杂的二阶优化，而是通过per-channel缩放和零点点校准，简单高效。不需要对权重做混合精度，就能达到和GPTQ相当甚至更好的精度，量化速度快一个数量级。

为什么AWQ工程上这么受欢迎？

GPTQ需要反向传播和Hessian计算，AWQ只需要前向过一遍校准数据做激活统计，然后直接计算scale。量化速度快，部署简单，kernel容易优化，精度还更好。

3.3 KV Cache 量化：长上下文的关键

很多人忽略了一个问题：随着上下文窗口变长，KV Cache的显存占用会超过模型本身。举个例子：70B模型FP16权重140GB，但在128k上下文长度下，每层KV Cache大小是2 * layers * heads * head_dim * seq_len * sizeof(fp16)，算下来也有几十GB到上百GB。

KV Cache量化是长上下文推理落地的必备技术，但它和权重量化有本质区别：KV是在线生成的，每个token都在追加，没有提前校准的机会；同时KV存在时间轴上累积，误差会影响后续所有token的生成。

KV Cache量化的特殊挑战

  • 异常值更极端：注意力机制的本质是softmax，会把概率集中到少数位置，导致这些位置的KV值特别大
  • 误差传播：第100个token的KV量化误差会影响到第10000个token的注意力计算
  • 需要逐token动态量化：不能离线预计算scale，必须在线算

KIVI ICML 2024

核心方案：Key用per-channel对称INT4量化，Value用per-token非对称INT4量化。对Key来说通道维度的离群值稳定，对Value来说token维度的分布更重要。

关键技巧：预留1-2个"重保位"用FP16存每个通道最大的几个离群值，其余值量化到4bit。2bit的额外开销换来巨大的精度提升。

StreamingQuant 2024

创新点：用在线算法增量更新量化参数，不需要等整个序列都生成完再量化。每个token到来时只用O(1)开销更新scale和zero-point，适合真正的流式生成场景。

3.4 Sub-4bit：2-3bit极低比特量化

4bit已经很成熟了，下一个战场是2-3bit。这个区间的量化难度陡增——4bit有16个level，靠线性缩放还能凑合；2bit只有4个level，3bit只有8个level，线性量化的信息损失已经不可接受，必须引入更高级的编码方式。

SpQR (Sparse Quantized Representation) NeurIPS 2023

核心思路：把GPTQ和稀疏性结合。识别出量化误差特别大的"敏感权重"，把它们单独挑出来用16bit存，其余权重量化到3-4bit。

效果：平均3.5-4bit就能达到接近FP16的精度，敏感权重只占总参数的不到0.1%。

QuIP (Quantization with Incoherence Processing) ICML 2024

核心洞察：权重矩阵的相干性（coherence）是导致低比特量化困难的根源——如果W的行和列高度相关，量化误差会集中到某些方向被放大。

方法：量化前先用随机正交矩阵对权重做"去相干"预处理，量化后推理时再乘回来。在2bit下就能达到非常惊人的精度。

AQLM (Additive Quantization for Language Models) ICLR 2024

突破：不把每个权重独立量化，而是把多个权重作为一个小组用codebook向量相加来表示——本质是加性量化（Additive Quantization），类似向量量化但支持多个码本相加。

效果：2bit下能达到和4bit GPTQ/AWQ相当的精度，是目前Sub-4bit精度最好的方案之一，但kernel实现复杂度高，推理加速还不够成熟。

3.5 量化感知训练 QAT：精度终极方案

PTQ（Post-Training Quantization，训练后量化）不需要重新训练模型，拿训好的模型直接量化，方便快速。但到了2-3bit这个级别，纯PTQ不可避免会有精度损失，这时候就需要QAT了。

QAT的思路很直接：在训练（微调）过程中就模拟量化噪声，让模型自己学会适应低比特表示。前向传播时用伪量化（fake quantize），反向传播还是用浮点更新权重。

⚠️ QAT的代价

QAT需要几百到上千个GPU小时做微调，数据准备和调参成本高。不是所有场景都值得。对于W8/W4，PTQ足够好；但如果要做到2bit甚至1bit，QAT几乎是必须的。

LLM-QAT（Meta, 2023）

Meta提出的方法：不需要大量标注数据，用模型自己生成的蒸馏数据做QAT就行。学生模型（量化版）学习教师模型（FP16版）的输出分布，而不是拟合硬标签。4bit下几乎无损，2bit也能work。

4. 硬件协同设计

量化算法如果没有硬件支持就只是纸面指标。真正快不快，最终要看kernel能不能在硬件上跑满算力。

硬件/框架 支持精度 特点
NVIDIA TensorRT-LLM INT4/INT8/FP8 官方优化最成熟，Hopper/Ada架构有FP8 Tensor Core
llama.cpp (ggml) Q2_K - Q8_0 CPU/Apple Silicon优化极致，k-quant系列非均匀量化是特色
Apple MLX INT4/INT8/FP16 苹果硅统一内存架构优化，M系列芯片效果出色
Qualcomm AI Engine INT4/INT8 手机端NPU支持，端侧部署主流方案
关于ggml的k-quant

llama.cpp里的Q4_K_M、Q5_K_S这些量化类型不是简单的线性INT量化，而是它独创的非均匀块量化——每32-64个权重作为一个block，block内部分权重用高比特存，部分用低比特存，根据重要性动态分配比特数。这是为什么llama.cpp的Q4_K_M看起来是"4bit"但实际精度比很多纯INT4方案好的原因。

5. 工程实践中的坑

Per-channel vs per-token vs per-group

量化粒度的选择直接决定精度和速度的trade-off：

  • per-tensor：整个权重矩阵共用一个scale，最快但精度最差
  • per-channel：每输出通道一个scale，W8A8常用
  • per-group：每128/64/32个权重共用一个scale，W4A16标准做法。group size越小精度越好但scale越多，内存开销和计算开销越大，一般128是甜点
  • per-token（激活）：每个token的激活动态算scale，必须在线计算，是激活量化的标准做法

哪些层对量化最敏感？

不是所有层量化到4bit都一样安全。实践中一般规律：

  1. 最敏感：LayerNorm的缩放因子、最后的lm_head输出层
  2. 中等敏感：QKV投影层、注意力输出投影
  3. 最不敏感：MLP的上下投影层，可以激进量化

很多部署框架会自动对敏感层跳过量化（保持FP16），不敏感层用4bit，用很小的精度代价换最大的压缩率。

校准数据怎么选？

PTQ方法都需要校准数据来统计激活分布。一个反直觉但正确的结论是：校准数据不需要和下游任务高度相关。用128-256条随机的预训练数据（比如wikitext的随机片段）就足够了，关键是数据分布要覆盖各种常见模式，而不是用任务相关的SFT数据。用SFT数据校准反而可能过拟合到特定任务，导致通用能力下降。

6. 未来方向与总结

大模型量化技术发展到今天，W8A8和W4A16已经是非常成熟的工业级技术，开源工具链完善，可以放心在生产中使用。当前研究热点集中在这几个方向：

  1. 2bit极限压缩：AQLM、QuIP等方案已经证明2bit是可行的，但还需要更好的kernel支持才能真正落地加速
  2. FP8统一量化：NVIDIA Hopper和AMD MI300都原生支持FP8，FP8不需要复杂校准，直接就能用，可能是下一代推理标准精度
  3. KV Cache 1bit/2bit量化：百万token长上下文的刚需，目前还在快速迭代
  4. 自适应量化：根据输入复杂度动态切换量化精度，难的问题用高精度，简单问题用低精度
  5. 训练时就设计成可量化：从预训练开始就考虑量化友好性，而不是训完再补救
📌 给实践者的建议

1. 如果是部署7B/13B/70B模型，优先试AWQ 4bit group=128，精度速度平衡最好
2. 如果有NVIDIA A100/H100/H200，追求极致速度，W8A8 + TensorRT-LLM是首选
3. 如果是CPU/苹果硅/端侧部署，用llama.cpp的Q4_K_M或Q5_K_M
4. 上下文长度超过32k，一定要做KV Cache量化，否则显存撑不住
5. 不要一开始就尝试2bit，先把W4调明白再说

量化不是一个孤立的技术点——它和算子优化、内存调度、硬件特性深度绑定。理解量化背后的原理，而不是只会调用现成的量化脚本，才能在实际问题出现时知道该从哪里调优。
